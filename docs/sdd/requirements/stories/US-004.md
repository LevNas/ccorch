# US-004: Multi-Layer Safety

## User Story

**As a** Claude Code user
**I want to** child panes to be safe even when running with `--dangerously-skip-permissions`
**So that** the risk of unrestricted agents performing destructive operations is minimized

## Acceptance Criteria (EARS Notation)

### Functional Requirements

- **REQ-004-001**: When launching the Main Brain (DEPTH=1), the system shall specify permitted tools via `--allowedTools`
- **REQ-004-002**: When launching a Child (DEPTH=2), the system shall specify `--allowedTools` equal to or more restrictive than DEPTH=1
- **REQ-004-003**: When launching a Grandchild (DEPTH=3), the system shall specify `--disallowedTools "Agent"` to structurally prohibit new pane creation
- **REQ-004-004**: When launching a pane at any depth, the system shall inject guard rail rules for that depth via `--append-system-prompt`
- **REQ-004-005**: If the Bash tool is permitted, the system shall apply pattern restrictions such as `Bash(git:status,git:diff,git:add,git:commit)`

## Depth-Based Tool Permissions

| Tool | DEPTH=1 | DEPTH=2 | DEPTH=3 |
|------|---------|---------|---------|
| Read | Yes | Yes | Yes |
| Edit | Yes | Yes | Yes |
| Write | Yes | Yes | Yes |
| Bash | Yes (restricted) | Yes (restricted) | Yes (restricted) |
| Grep | Yes | Yes | Yes |
| Glob | Yes | Yes | Yes |
| Agent | Yes | Yes | **No** |

## Guard Rail Prompt Examples

```
DEPTH=1: "You are a CCORCH Main Brain (DEPTH=1). You can delegate tasks to child panes.
Destructive git operations (push --force, reset --hard) are prohibited."

DEPTH=3: "You are a CCORCH Grandchild (DEPTH=3). You must not create new panes.
Execute only your assigned task."
```
